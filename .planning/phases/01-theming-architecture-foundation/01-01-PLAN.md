## Tailwind config transformation shape (RESEARCH §4, PATTERNS Edit #2 AFTER)

- `accent.DEFAULT` -> `--accent-rgb`
- `accent.hover`   -> `--accent-hover-rgb`
- `accent.muted`   -> `--accent-muted-rgb`

- `surface['dark-card']`-> `--surface-dark-card-rgb`
- `ink` -> `--text-rgb`
- `slate.50` through `slate.900` -> `--neutral-50-rgb` through `--neutral-900-rgb`
- `fontFamily.sans`    -> `var(--font-body)`
- `fontFamily.display` -> `var(--font-display)`

direct Tailwind config consumer in Phase 1 — they tokenize Category B hex usages that Phase 2
will migrate. They are declared now so Phase 2 doesn't need a contract change.

## FOUC script shape (RESEARCH §3, locked by D-12)

```html
<script>
    (function () {
        var p = new URLSearchParams(location.search).get('client');
        if (p) document.documentElement.setAttribute('data-theme', p);
    })();
</script>
```

- IIFE wrapper (PATTERNS S2) keeps `p` out of `window.*` since the rest of `<head>` shares global scope.
- 4-space indent matching the rest of `<head>` (CONVENTIONS.md).
- `var` not `const` — avoids any temporal-dead-zone edge case during early `<head>` parse.
- No try/catch, no DOMContentLoaded wait, no localStorage — all explicitly rejected (RESEARCH §3, Deferred Ideas).
- Variable name `p` is intentionally short (the script is short; D-11's non-engineer prose principle applies to the recipe comment, not to this script).

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| URL query string → DOM attribute | The `?client=<slug>` URL param flows into `document.documentElement.setAttribute('data-theme', slug)` via the FOUC script. This is the only untrusted input that crosses into DOM state in Phase 1. |
| DevTools console → CSS custom properties | A user can call `document.documentElement.style.setProperty('--accent-rgb', '...')` from DevTools. This is the intended testing affordance (V-2 in VALIDATION.md), not a real threat surface. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-01 | T (Tampering) — input | FOUC `<script>` reading `?client=<slug>` | mitigate | Use `setAttribute('data-theme', p)` directly with the raw string — NOT `innerHTML`, NOT CSS string concatenation, NOT `eval`. Worst case: a malicious URL like `?client=<script>alert(1)</script>` lands the literal string `<script>alert(1)</script>` as the `data-theme` attribute value. No CSS rule matches that attribute (Phase 1 has zero `[data-theme="..."]` blocks; any future client slugs use plain alphanumeric/dash form per D-10), so the page silently falls back to `:root` defaults per D-03 / D-15. The browser does not interpret `data-theme` attribute values as anything other than a string. No XSS vector exists. |
| T-01-02 | I (Information Disclosure) — none | n/a | accept | The app has no user data, no auth, no network calls beyond Google Fonts CDN + Tailwind CDN. Nothing to disclose. |
| T-01-03 | D (Denial of Service) — load order | Misordered `<head>` elements (token `<style>` after CDN script) | mitigate | Per RESEARCH Pitfall 2 + §5 ordering. Task 3 explicitly moves the CDN script to AFTER the token `<style>`. Verification (V-2, V-5) catches the failure mode in browser. Not a security threat per se — but the only way Phase 1's contract can "break" a user's experience. |
| T-01-04 | E (Elevation of Privilege) — n/a | n/a | accept | Single-page static app, no auth model, no privilege boundaries to elevate across. |
| T-01-05 | S (Spoofing) — workshop links | Shared `?client=<slug>` URL semantics | accept | A user can share a URL with any `?client=` slug. For Phase 1, no override blocks exist, so all slugs render the same identity. For future phases, the trust model is: any slug that matches a defined override renders that theme; everything else falls back silently to Overdrive (D-03). No identity, no authentication, no spoofing surface. |
| T-01-06 | R (Repudiation) — n/a | n/a | accept | No actions are logged, no state is mutated server-side, no audit trail required. The app is a deterministic static read. |

**Net assessment:** Phase 1's only real threat is T-01-01 (URL-param tampering), and it is mitigated by using `setAttribute` with a raw string (the universally safe DOM pattern for untrusted strings landing as attribute values). The other STRIDE categories do not apply meaningfully to a single-file static app with no auth, no PII, no persistence, no server.
</threat_model>
